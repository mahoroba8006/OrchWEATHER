## 1. 文書概要

### 1.1 目的
本書は Orch.Weather が「誰に・何を・どう提供するか」を定義する。既存実装からのリバースエンジニアリングにより、ユーザー要件（機能・動作）とシステム要件（利用環境）を網羅する。

### 1.2 用語
| 用語 | 意味 |
|------|------|
| 地点（Location） | 気象データを表示する対象の緯度経度＋名称。Firestore に保存する。 |
| 現在地（`__geo__`） | GPS で取得した一時的な地点。Firestore に保存しない仮想地点。 |
| ティア | ログイン状態・AI許可の有無で決まる機能制限レベル（→ 3章）。 |
| AI許可（`aiAllowed`） | サーバー側の許可名簿に含まれるかで決まる、AI機能・地点上限拡張・CSV出力の解錠フラグ。 |
| ゲストモード（`guestMode`） | 未ログインのまま現在地のみで試用する状態。localStorage に永続。 |
| GDD / 積算温度 | Growing Degree Days。基準温度を超えた分の日々の積み上げ。作物の生育指標。 |
| 飽差（VPD相当） | 飽和水蒸気量−実水蒸気量 [g/m³]。施設園芸の環境管理指標。 |

## 2. システム概要

### 2.1 コンセプト
農家（特に施設園芸・露地栽培）向けの**気象データ可視化・活用アプリ**。「予報の提供」ではなく「データの表示・活用支援」に位置づけ、気象業務法上の予報業務に該当しない表現・機能設計とする（→ 5.5 法令要件）。

### 2.2 提供価値（3つの柱＝3タブ）
| タブ | 内部ID | アイコン | 提供価値 |
|------|--------|----------|----------|
| 空もよう | `weather` | Sun | 現在〜15日先の気象を、時間別・日別・リスク・AIアドバイスで把握する |
| 空くらべ | `analysis` | BarChart2 | 過去の年・地点を比較し、積算温度などで生育の進み具合を読む |
| 空しらべ | `history` | Clock | 任意期間の過去実績データを表・グラフで確認・エクスポートする |

補助画面: **設定**（`settings`）、**使い方**（`help`）。

### 2.3 利用者
農業従事者を主対象とする一般消費者向け Web アプリ。招待制ではなく一般公開。氏名・作物などの個人プロファイルは要求しない（Google アカウントのみ）。

## 3. 利用者区分とアクセス権（ティア）

ログイン状態と AI 許可（`aiAllowed`）の組み合わせで4区分する。区分は `store.ts` の `user` / `guestMode` / `aiAllowed` で判定する。

| # | 区分 | 判定条件 | 地点登録 | 地点上限 | AIコメント | CSV出力 | データ永続化 |
|---|------|----------|----------|----------|-----------|---------|-------------|
| T0 | 未ログイン | `user===null && !guestMode` | ― | ― | ― | ― | ― （ランディングページのみ表示） |
| T1 | ゲスト | `!user && guestMode` | 不可 | ― | 不可 | 不可 | なし（現在地のみ・保存されない） |
| T2 | ログイン・AI非許可 | `user!=null && aiAllowed===false` | 可 | **10件** | Coming Soon カード表示 | 不可 | Firestore に保存 |
| T3 | ログイン・AI許可 | `user!=null && aiAllowed===true` | 可 | **50件** | 実生成（Gemini） | 可 | Firestore に保存 |

- **地点上限:** `store.addLocation` で `aiAllowed ? 50 : 10`。超過時は例外 `登録できる地点は最大N件までです`。
- **AI許可の決定:** サーバー（`/api/me`）が Firebase ID トークンを検証し、環境変数 `AI_ALLOWLIST`（許可メールのカンマ区切り名簿）に含まれるかで `{aiAllowed}` を返す。クライアントはこの1フラグで機能を分岐する。
- **CSV出力:** `aiAllowed` のみ許可（`DailyRawTable`）。
- **ログアウト時:** `resetUserData()` で前ユーザーの `locations` / `userSettings` / `aiAllowed` をクリアし、共有ブラウザでの情報残留を防ぐ。

## 4. ユーザー要件（機能要件）

機能IDは `UR-<領域>-<連番>` で付与する（更新時のトレーサビリティ用）。

### 4.1 認証・オンボーディング

| ID | 機能 | 動作 |
|----|------|------|
| UR-AUTH-01 | Google ログイン | ランディングページ／ログイン画面から Google アカウントでサインイン。iOS Safari は `signInWithRedirect`、それ以外は `signInWithPopup`（PWA でポップアップが機能しない問題への対応）。 |
| UR-AUTH-02 | リダイレクト結果処理 | 起動時に `getRedirectResult` を評価し、リダイレクト後の認証完了・エラーをサイレント処理。 |
| UR-AUTH-03 | ゲスト試用 | ランディングページ「ログインせずに試す」で `guestMode=true`。現在地のみでアプリ本体に入る。 |
| UR-AUTH-04 | ログアウト | ヘッダー／設定からログアウト。前ユーザーデータをクリア。 |
| UR-AUTH-05 | 起動フリーズ防止 | 認証・Firestore ロードを try/catch で保護し、失敗時も必ず `authLoading=false` にして無限ローディングを防ぐ。 |
| UR-AUTH-06 | 初回ユーザードキュメント作成 | 初ログイン時、Firestore にユーザードキュメントが無ければ作成（存在すれば作らない）。 |

### 4.2 地点管理（設定→地点設定）

| ID | 機能 | 動作 |
|----|------|------|
| UR-LOC-01 | 現在地登録 | GPS（Geolocation API, `enableHighAccuracy:false, timeout:10s`）で現在地を取得し地点化。 |
| UR-LOC-02 | 地図から選択 | Leaflet 地図（OpenStreetMap タイル）上のクリックで座標を選び登録。GSI 逆ジオコーディングで地名候補を取得。 |
| UR-LOC-03 | 手動追加 | 名称・緯度・経度を手入力して登録。 |
| UR-LOC-04 | 地図で修正 | 既存地点を地図で微調整。編集時は登録座標を維持（現在地へ飛ばさない）。新規時のみ GPS 追従。 |
| UR-LOC-05 | 編集・削除 | 登録地点の名称・座標変更、削除。 |
| UR-LOC-06 | 気象庁エリア紐付け | 登録時に緯度経度から気象庁 class20s エリアコード（7桁）を解決し `jmaAreaCode` として保持。注意報・警報の取得に使用。 |
| UR-LOC-07 | デフォルト地点 | 起動時に最初に表示する地点を指定。無効・未指定なら現在地を自動取得。 |
| UR-LOC-08 | 上限制御 | T2=10件 / T3=50件。超過時はエラー表示。 |

### 4.3 空もよう（`weather` タブ）

現在〜15日先の気象を表示する。地点セレクタ・「現在地を表示」ボタン・最終更新時刻・手動更新ボタンを上部に持つ。

| ID | 機能 | 動作 |
|----|------|------|
| UR-WX-01 | 地点切替 | 現在地（📍）＋登録地点をドロップダウンで切替。 |
| UR-WX-02 | 手動更新／最終更新表示 | 「更新」ボタンで再取得。最終取得時刻（JST HH:MM）を表示。 |
| UR-WX-03 | 日別予報 | 今日〜14日後（**15日分**）を表示。各日を午前（4〜12時）／午後（12〜20時）／夜間（20〜翌4時）に3分割し、天気アイコン・降水確率・気温・風などを表示。 |
| UR-WX-04 | 天気コードモード切替 | 「リスクでみる」＝時間帯で最も深刻な天気を表示 / 「概況でみる」＝最も頻度の高い天気を表示。設定は保存。 |
| UR-WX-05 | 欠損日の扱い | 気温データ未作成の日は 0℃ と誤表示せず「—」表示（`isPlaceholder`）。 |
| UR-WX-06 | 半日クリック→時間別ジャンプ | 日別の午前/午後/夜間セルをタップすると、対応時刻の時間別テーブルへスクロール。 |
| UR-WX-07 | 時間別予報テーブル | 約10日分（72時間を主対象）の時間別データ（気温・露点・湿度・飽差・風速・風向・瞬間風速・降水量・降水確率・日射・UV・降雪・気圧・天気）を横スクロールテーブルで表示。ミニグラフ・降水の体感ラベル・日の出/入り・夜間網掛け・警報ガントバーを含む。 |
| UR-WX-08 | 注意報・警報サマリー | 発令中の気象庁注意報・警報を一覧表示（→ 4.7）。 |
| UR-WX-09 | AIステータスバー | AI生成中は「空のようすをみています…」、完了後は要約バーを表示しAIカードへスクロール。 |
| UR-WX-10 | AI農作業コメント | 予報・警報が揃うと AI コメントを非同期生成・表示（→ 4.6）。T2 は Coming Soon カード。 |

### 4.4 空くらべ（`analysis` タブ）

過去データの年・地点比較。最大2件（基準＋比較）を並べ、7チャートで表示する。

| ID | 機能 | 動作 |
|----|------|------|
| UR-CMP-01 | 比較対象選択 | 地点×年を最大2件選び「表示」ボタンで取得（選択変更だけでは取得しない）。「📍現在地」を常時選択肢に持つ。 |
| UR-CMP-02 | 7チャート | 気温／降水量／積算温度（GDD）／日射量／日照時間／湿度／飽差 を切替表示。 |
| UR-CMP-03 | 日次／月次モード | 日次（日別プロット）と月次（月平均プロット）を切替。 |
| UR-CMP-04 | 10日予報オーバーレイ | 今年の対象には予報を破線で重畳（気温・湿度・飽差＝破線縦バー、降水・日照・日射・積算温度＝破線累積線）。 |
| UR-CMP-05 | 累積系の差分（Δ） | 累積指標で「基準の現在値に、比較年が到達した日」を逆引きし、Δ日数を表示。 |
| UR-CMP-06 | 累積開始日設定 | 降水・日照・日射・GDD それぞれの累積開始日（MM-DD）をユーザー設定可能。 |
| UR-CMP-07 | 基準温度設定 | GDD 計算の基準温度を2つ保持（例 10℃ / 3.5℃）、選択して使用。 |
| UR-CMP-08 | モバイル操作 | ドラッグでパン、タップで値表示、crosshair（縦横点線＋日付ラベル）表示。初期表示は前々月〜翌月。 |

### 4.5 空しらべ（`history` タブ）

任意期間の過去実績を表・グラフで確認する。

| ID | 機能 | 動作 |
|----|------|------|
| UR-HIS-01 | 期間指定表示 | 過去の任意期間の日別・時間別データを表示（Open-Meteo Archive / Historical Forecast API を期間で切替）。 |
| UR-HIS-02 | データ源の自動切替 | 2022年以降は Historical Forecast API、それ以前は Archive API を使用。取得できない項目（降水確率・0℃層高度・UV等）はデータ駆動で行を非表示。 |
| UR-HIS-03 | 日別生データ表 | 全指標×日数の一覧表（`DailyRawTable`）。最高/最低ベース10列。 |
| UR-HIS-04 | CSV出力 | UTF-8 BOM 付き CSV をダウンロード（**T3限定**）。 |

### 4.6 AI 気象・農作業コメント（横断機能）

Cloudflare Pages Function 上で Gemini により生成。設定→空のアドバイスでセクションを選択。

| ID | 機能 | 動作 |
|----|------|------|
| UR-AI-01 | 標準4セクション | 空ごよみ（概況）／畑しごと（一般外作業）／散布どき（防除）／施肥どき／天気の備え 等を自然文で生成（各150〜180字）。 |
| UR-AI-02 | 表示セクション選択 | 表示する AI セクションをユーザーが選択（保存）。 |
| UR-AI-03 | カスタマイズAI | ユーザー入力プロンプト（最大200字）で独自コメントを生成（オプトイン）。 |
| UR-AI-04 | キャッシュ | 入力ハッシュ単位で Firestore にキャッシュ（TTL 4時間）。標準タブとカスタムはキー接頭辞で分離。 |
| UR-AI-05 | サーバー認可 | AI エンドポイントは Firebase ID トークン検証＋許可名簿で保護（401/403）。 |
| UR-AI-06 | ガードレール | 農業・気象以外は回答拒否。気象予報値はデータ厳守、農学的一般知識（乾き時間等）は一般論として使用可。命令形禁止・提案形限定・専門用語言い換え。 |

### 4.7 注意報・警報（横断機能）

気象庁防災情報 JSON（r8）を直接取得して表示。

| ID | 機能 | 動作 |
|----|------|------|
| UR-JMA-01 | 発令中警報の取得 | 地点の `jmaAreaCode` から都道府県コードを解決し r8 JSON を取得。 |
| UR-JMA-02 | 有効性フィルタ | 終了時刻切れ・解除・古い電文を除外（1箇所 `fetchJmaWarnings` に集約）。 |
| UR-JMA-03 | 表示グループ制御 | 17グループ（大雨・土砂災害・洪水・大雪・強風・風雪・波浪・高潮・乾燥・霜・低温・雷・濃霧・なだれ・融雪・着氷・着雪）の表示ON/OFFを設定。特別警報・未分類は常時表示。 |
| UR-JMA-04 | ステータス表示 | 発表／継続／更新／特別警報を区別して表示。 |
| UR-JMA-05 | ガントバー | 日別・時間別に有効期間を帯（ガントバー）で可視化。r8 は終了時刻を持たないため `now+12h` まで＋アロー表示。 |

## 5. 非機能要件

### 5.1 性能
- 起動時の白画面・無限ローディングを排除（try/catch＋確実な `authLoading=false`）。
- 気象データ取得は非同期。予報・警報・AI は独立して取得し、揃った順に描画。
- AI はキャッシュ（TTL 4h）でコスト・レイテンシを抑制。

### 5.2 可用性・障害耐性
- 起動時 JS エラーは `index.html` の `window.onerror`/`unhandledrejection` で捕捉し、白画面ではなくエラーメッセージを表示。
- 外部 API 障害時はエラーメッセージ表示にとどめ、アプリ全体はクラッシュさせない。

### 5.3 セキュリティ
- Firestore はユーザー本人のドキュメントのみ読み書き可（`request.auth.uid == userId`、デフォルト拒否）。
- AI エンドポイントはサーバー側でトークン検証＋許可名簿。Gemini キーはサーバーサイドのみ。
- 全ルートにセキュリティヘッダ＋CSP（→ 6.6）。
- Firebase 公開キー・Gemini キーは別 GCP プロジェクト。公開キーは HTTP リファラー制限。

### 5.4 プライバシー
- 収集するのは Google アカウント情報（表示名・メール・アイコン）と登録地点・設定のみ。
- GA4 の計測項目・目的をプライバシーポリシーに明記。Cookie 同意バナーは設けない（ポリシー記載方式）。

### 5.5 法令要件（気象業務法）
- 「天気予報」「気象予測」等の予報業務を示す表現を避け、「データを表示／活用」に統一。
- 免責文で「シミュレーション・サポートツール」であることを明示。
- リスク検出・注意情報は気象庁の発表・実データに基づく可視化にとどめ、独自予報を生成しない。

### 5.6 UX・対応環境
- モバイルファースト（ボトムナビ3ボタン、safe-area 対応、プルリフレッシュ無効）。
- PWA（manifest、standalone、apple-touch-icon）。
- デザイン方針「自然と水」（ミントグリーン＋水のグラデーション、ガラスモーフィズム、Outfit フォント）。

## 6. システム要件

### 6.1 クライアント実行環境
| 項目 | 要件 |
|------|------|
| 対象 | モダンブラウザ（Chrome / Safari / Firefox / Edge）。ビルドターゲット `es2020`。 |
| デバイス | スマートフォン・タブレット・PC（レスポンシブ）。iOS Safari を実機考慮。 |
| 必須API | Geolocation（現在地）、localStorage（ゲストモード永続）。 |
| 表示言語 | 日本語（`<html lang="ja">`）。タイムゾーンは Asia/Tokyo 前提。 |

### 6.2 技術スタック
| 層 | 技術 |

| Open-Meteo Forecast API | 現在〜予報の時間別・日別 | 不要 |
| Open-Meteo Archive API | 過去実績（〜2021年含む長期） | 不要 |
| Open-Meteo Historical Forecast API | 2022年以降の過去実績 | 不要 |

| OpenStreetMap タイル | 地図表示 | 不要 |

### 6.5 環境変数
**クライアント（Vite `VITE_*`＝ビルド時にJSへ焼き込み）:**
| 変数 | 意味 |

### 6.6 セキュリティ構成
- **`public/_headers`（全ルート）:** `X-Frame-Options: DENY` / `X-Content-Type-Options: nosniff` / `Referrer-Policy` / `Strict-Transport-Security`（HSTS 1年） / `Permissions-Policy`（geolocation=self、他は無効） / `Content-Security-Policy`。
- **CSP の許可先:** アプリが実際に通信する先だけを許可（Open-Meteo 3種、JMA、GSI、OSM タイル、Meteocons CDN、Google 認証・Analytics・Fonts、Firebase）。通信先の増減時は必ず `_headers` も更新する。
- **Firestore ルール:** リポジトリ外（Firebase コンソール管理）。本人限定＋デフォルト拒否。

## 7. 前提条件・制約条件（Assumptions & Constraints）

設計判断の「隠れた根拠」を明文化する。これらが崩れると設計そのものが成立しない、または再検討が必要になる。

### 7.1 前提条件（Assumptions）
| # | 前提 | 崩れた場合の影響 |
|---|------|------------------|
| A-01 | 対象ユーザーは日本国内の農業従事者（施設園芸・露地） | 飽差の g/m³ 採用、日本語UI、JST固定などの前提が崩れる |
| A-02 | タイムゾーンは Asia/Tokyo 固定 | 日付・時間帯（午前/午後/夜間）集計、UTC基準日付計算の全ロジックが再設計 |
| A-03 | 単独開発・単独保守 | PMBOK的なチーム運用・調達・要員計画は不要（本設計書もこの前提で軽量化） |
| A-04 | 気象データは Open-Meteo（ERA5 / JMA-MSM の無料公開データのラッパー）に依存 | データ源変更時は空くらべの単一ソース統一制約に注意（→ C-05） |
| A-05 | クライアントは Geolocation API・localStorage を利用可能 | 現在地機能・ゲストモードが機能しない（フォールバック要） |
| A-06 | 利用者は Google アカウントを保有 | ログイン導線が成立しない（他IdP未対応） |

### 7.2 制約条件（Constraints）
| # | 制約 | 出所・理由 |
|---|------|-----------|
| C-01 | 気象業務法上の「予報業務」に該当しない表現・機能に限定する | 個人段階では予報業務許可の費用対効果が低い。「データを表示／活用」に統一、独自予報を生成しない |
| C-02 | Open-Meteo の商用利用は有料プラン必須（Free tier 不可） | 規約はアプリ単位で判定。AIサブスクを持つ＝商用。天気を無料で出しても有料プランは不可避 |
| C-03 | 空くらべ（年比較）は Archive/Historical API に依存し、上位プラン（Standard/Professional級）が必要 | 年跨ぎが本質でStandardのForecast枠では代替不可。必要プランは Open-Meteo へ要照会（未確定） |
| C-04 | 外部コスト上限は Gemini（従量）・Open-Meteo（固定）・Firebase/Cloudflare（無料枠＋従量）に規定される | 無料解放はコール枠を収益ゼロユーザーに開放する＝固定費増。archive永続キャッシュが実質前提 |
| C-05 | 空くらべの比較データは全年で単一データ源に統一する | モデル差が気温差として現れ比較が破綻するため、ハイブリッド不可 |
| C-06 | `VITE_*` 環境変数はビルド時に焼き込まれる | 値変更・追加は再デプロイ（リビルド）必須。名前不一致は無言で機能停止 |
| C-07 | Firestore セキュリティルールは Firebase コンソールで管理（リポジトリ外） | バージョン管理外のため誤操作に気づきにくい（リポジトリ化を推奨） |
| C-08 | develop ステージングは本番と同一 Firestore を共有する | 専用DBがない。リハーサル時の書き込みは本番データに混じりうる |
| C-09 | 自動テスト基盤なし（vitest 未導入） | 検証は `npm run build` 通過＋手動・実機確認による（TDD不適用） |
| C-10 | Cloudflare Smart Placement は無効 | archive-api への経路問題対策。日本PoP固定を避ける意図 |
| C-11 | CSP はアプリの実通信先のみ許可 | 通信先を増減したら `public/_headers` を必ず更新する |

### 7.3 外部依存（Dependencies）
本システムは以下の外部サービスの稼働・規約に依存する。各々の障害・規約変更・料金改定はリスクとして [リスク登録簿](05-risk-register.md) で管理する。

Open-Meteo（Forecast / Archive / Historical Forecast）、気象庁 防災情報 JSON、国土地理院 逆ジオコーディング、OpenStreetMap タイル、Firebase（Auth / Firestore）、Google Gemini API、Google Analytics 4、Cloudflare Pages / Functions。
