## ADR-0001: フロントエンド技術スタック（React 19 + Vite + TypeScript + Zustand）
**状態:** 実装済み
**文脈:** 単独開発の SPA。高速なローカル開発と型安全、モバイル対応が要る。
**決定:** React 19 + TypeScript + Vite 8。UIは Recharts（グラフ）、lucide-react（UIアイコン）、Leaflet（地図）。ビルドターゲット `es2020`。
**理由:** Vite の高速 HMR、React 19 の最新機能、TS による型安全。Recharts は宣言的でカスタム描画（`Customized`・カスタムバー）も可能。
**結果:** バンドルは単一チャンクで 1.4MB（gzip 420KB）＝コード分割の余地あり（警告のみ・実害許容）。iOS Safari 対応のため `es2020` 明示・起動エラーハンドラを `index.html` に配置。

## ADR-0002: バックエンドは Cloudflare Pages Functions（Firebase Functions 不採用）
**状態:** 実装済み
**文脈:** AIコメント生成のためサーバー側で Gemini キーを秘匿し、トークン検証をしたい。
**決定:** ホスティングとサーバー処理を Cloudflare Pages + Pages Functions（`functions/api/*`）に統一。Firebase Functions は使わない。
**理由:** 静的配信とサーバー関数を同一プラットフォームで完結。Cloudflare 強制ゆえ、後の課金フェーズでもカスタムクレーム移行（A2）が Firestore REST 配管（A1）より軽くなる。
**結果:** `compatibility_date=2024-09-23`。Smart Placement は無効（archive 経路問題対策・[ADR-0005](#adr-0005) 関連）。エンドポイント: `ai-comment` / `ai-custom` / `me` / `archive`（休眠）/ 共有ガード `_auth`。

## ADR-0003: 認証・DB は Firebase（Google 認証のみ＋Firestore）
**状態:** 実装済み
**文脈:** 個人開発でユーザー認証と地点・設定の永続化が要る。想定ユーザーは Google アカウント保有前提。
**決定:** Firebase Authentication（Google プロバイダのみ）＋ Cloud Firestore。
**理由:** 認証・DB・ホスティング補完が揃い、個人規模で運用が軽い。他IdPは当面不要。
**結果:** iOS Safari は `signInWithRedirect`、他は `signInWithPopup`（PWAでポップアップが機能しない問題への対応）。Firestore 構造は `/users/{uid}` ＋サブコレクション `locations` / `aiComments`。

<a id="adr-0004"></a>
## ADR-0004: 状態管理は Zustand・persist なし・Firestore 直同期
**状態:** 実装済み
**文脈:** グローバル状態（user / locations / userSettings / geo / aiAllowed / guestMode）を管理する。
**決定:** Zustand。localStorage への persist ミドルウェアは使わず、真実の源は Firestore とする（ゲストモードのみ localStorage）。
**理由:** persist と Firestore の二重管理は不整合の温床。真実の源を1つ（Firestore）に絞る。
**結果:** ストアの更新アクションは「Firestore 書き込み → 成功後にローカル state 更新」の順。ログアウト時 `resetUserData()` で前ユーザーの locations/userSettings/aiAllowed をクリア（共有ブラウザの情報残留防止）。

<a id="adr-0005"></a>
## ADR-0005: 気象データ源は Open-Meteo（3API）＋年で自動切替・比較は単一ソース統一
**状態:** 実装済み
**文脈:** 現在〜予報、および過去実績（長期含む）が要る。
**決定:** Open-Meteo の3API を用途で使い分ける。Forecast（現在〜予報）、Archive（〜2021年含む長期実績）、Historical Forecast（2022年〜）。空しらべ・空くらべは年（境界2022年）で自動切替。
**理由:** Open-Meteo は ERA5 / JMA-MSM 等の無料公開データのラッパーで無料・高品質。Historical Forecast は 2022年以降の再解析、Archive はより長期。
**結果（重要な制約）:** **空くらべ（年比較）は全年を単一データ源に統一しなければならない。** モデル差が気温差として現れ比較が破綻するため、年ごとに別モデルを混ぜるハイブリッドは不可（→ 制約 C-05）。モデル: Archive は year≥2016 が `jma_msm`（5km投影格子）、year<2016 が `era5_land`（0.1度規則格子）。

## ADR-0006: Open-Meteo 商用は Free tier 不可・空くらべは上位プラン前提
**状態:** 採用（制約）／照会未完
**文脈:** 有料化を見据える。Open-Meteo の規約と料金プランを精査した。
**決定:** 商用化する以上、Open-Meteo は有料プラン（Standard/Professional 級）を契約する前提で設計する。
**理由:** 規約逐語では非商用＝「サブスク・広告を持たないアプリ」。判定は**アプリ単位**。「天気は無料・AIだけ課金」でもアプリがAIサブスクを持てば商用＝Free tier 違反。空くらべは archive 依存で上位プランが要る。
**結果:** 必要プランが Standard（€29/¥4,500）か Professional（€99/¥15,000）かは3ソースが矛盾し**未確定**。損益分岐が3倍変わるため `info@open-meteo.com` 照会が最重要ペンディング。無料解放は有料コール枠を収益ゼロユーザーに開くため、[ADR-0007](#adr-0007) の永続キャッシュが実質前提。

## ADR-0007: archive 永続キャッシュ＝レスポンスの実格子点座標をキーにする2層方式
**状態:** 設計のみ（未実装）
**文脈:** 空くらべ等で archive を叩くコストを近隣ユーザー間で共有・削減したい。archive 履歴は完全不変でキャッシュ適性が高い。
**決定:** 自分で格子を切らず、Open-Meteo が**レスポンスで返す実格子点座標**（`latitude`/`longitude`）を正規化キーにする2層方式。1層目=粗グリッド丸め座標→格子点座標の探索表、2層目=格子点座標+year→気象データ。
**理由:** ナイーブな「0.05度丸め」は誤り。自作グリッドと Open-Meteo のモデル格子（特に JMA-MSM の5km投影格子）は原点も間隔も一致せず、同一セル内でも別格子点にスナップして違う値を返しうる。実格子点座標をキーにすれば、誤データ混入は構造的にゼロ。1層目が境界で振り分けをミスしても起きるのは「キャッシュミス＝再フェッチ」だけで**安全側に倒れる**。
**結果:** 現行コードは `daily` だけ読みレスポンスの格子点座標を捨てている（拾えば流用可）。未決: 保存先（Firestore か Cloudflare KV）、キャッシュ層の位置（クライアント側か Function 側か＝書き込み信頼境界・スタンピード対策）、粗グリッド粒度。

## ADR-0008: 注意報・警報は気象庁 防災情報 JSON（r8）を直接取得
**状態:** 実装済み
**文脈:** 地点の注意報・警報を表示する。
**決定:** 気象庁の防災ページが使う `warning/data/r8/{prefCode}.json` を直接取得する。フィルタ（有効性判定）は `fetchJmaWarnings` の1箇所に集約。
**理由:** 旧API（`warning/data/warning/`）は2026-05-28以降更新停止し全件「解除」で凍結。r8 が現行。フィルタを1箇所に集約することで UI・AI・ガントバーを単一ソースで制御し、基準ズレによる不整合を防ぐ。
**結果:** r8 は終了時刻を提供しない。終了時刻ありは `endMs<now` で除外、なし（濃霧・乾燥等）は発表時刻から6時間超で除外。levels が種別により object 形式になるバリアントに形式ガードあり（[lessons 2026-06-01](../../tasks/lessons.md)）。北海道・鹿児島・沖縄は prefCode 法則が崩れるため `AREA_TO_PREF` テーブルで個別マッピング。

## ADR-0010: 飽差は g/m³ を採用（kPa 不採用）
**状態:** 実装済み
**文脈:** 飽差には2定義がある。kPa（VPD, 国際標準）と g/m³（飽差, 日本の施設園芸現場標準）。
**決定:** **g/m³** を採用する。表示は小数1桁。
**理由:** 対象ユーザー（施設園芸農家）は現場の感覚値（推奨3〜7 g/m³）で管理する。kPa の推奨値（0.4〜1.0）は日本農業現場で馴染みが薄い。
**結果:** 計算式 `216.67 × e_s(T) / (T+273.15) × (1−RH/100)`、`e_s(T)=6.1078×10^(7.5T/(T+237.3))`（Tetens式）。`App.tsx` の `calcVPD`。将来 kPa 併記が要れば `calcVPD_gm3`/`calcVPD_kPa` に分割する。

## ADR-0011: 日付計算は UTC 基準 `addDays` に統一
**状態:** 実装済み（教訓由来）
**文脈:** 日別の夜間降水量が時間別合計と一致しない障害（6/3未明の雨が2日ズレて6/1夜間に合算）。
**決定:** カレンダー日付の加減算は必ず UTC 基準で行う。`src/lib/dateUtils.ts` の `addDays()`（`Date.UTC`＋`getUTCDate`）を使う。
**理由:** `new Date(dateStr+'T00:00:00')` はローカル（JST）解釈され、`toISOString()` で UTC に戻すと9時間戻って日付が1日ズレる。`setDate(-1)` と合わせ計2日ズレた。
**結果:** ローカル時刻 Date を生成して `toISOString()` で日付を取り出す処理は**禁止**。`forecast.ts` の夜間集計（0-3時→前日）と `DailyForecast.tsx` が共有 `addDays` を使用。

## ADR-0012: 天気アイコンは Meteocons v2 形式限定
**状態:** 実装済み（教訓由来）
**文脈:** WMOコード3（曇り）を `overcast`→`cloudy` に変えたらモバイルで曇りアイコンがリンク切れ表示になった。
**決定:** 天気アイコンは Meteocons の **v2形式（viewBox `0 0 512 512`・`symbol`/`use`/`animateTransform` 構造）** のみ使う。
**理由:** v3形式（viewBox `0 0 128 128`・`path` 直書き＋`clip-path` 参照）はモバイルの `<img>` 文脈で描画失敗する。clip-path 除去等の修正でも根本解決せず、CDN公式版でも同様＝構造自体の問題。
**結果:** WMOコード3は `overcast` に差し戻し。新規アイコン追加時は viewBox=512 の v2 であることを必ず確認。v3しか無ければ v2 に作り直すか代替を使う。

## ADR-0013: 天気コードモード severity/frequency はクライアント計算・キャッシュキーから除外
**状態:** 実装済み
**文脈:** 空もようの日別ブロックで「リスクでみる（最も深刻）」「概況でみる（最も頻度が高い）」を切替表示する。
**決定:** 2モードの差は `selectCode()` によるクライアント側計算のみとし、API レスポンスは同一。`useForecast` のキャッシュキーは `${lat},${lon}` とし **mode を含めない**。モード設定は Firestore に永続。
**理由:** モードをキャッシュキーに含めると切替のたびに Open-Meteo を再フェッチ（1〜3秒）してしまう。生データ（hourlyコード配列）を保持し `selectCode` の適用を分離すれば即時切替できる。
**結果:** タイブレークは `wmoSeverity.ts` の深刻度スコア（26段階）で決める。

<a id="adr-0014"></a>
## ADR-0014: AI モデルは Gemini 2.5 Flash（flash-lite 却下）＋文字数短縮
**状態:** 実装済み
**文脈:** AIコメントのコスト削減施策を検討（施策A/B/C）。
**決定:** モデルは `gemini-2.5-flash` を維持（`thinkingBudget:1024`）。flash-lite への切替（施策B）は**却下**。ただし出力文字数は 250-350→200-300字へ短縮。
**理由:** flash-lite は品質が悪くユーザー判断で不採用。コスト主成分は出力（thinking込み・出力単価は入力の約8倍）なので、モデルを落とさず文字数短縮で出力側を削るのが費用対効果が高い。1コール ≈ ¥1.2（確度70%）。
**結果:** `thinkingBudget` は品質直結のため当面触らない（唯一残るコストレバー）。項目別モデル分けは防除を別コールに分離する改修が要るため初手は単一コール1モデル。

## ADR-0015: AI キャッシュキー＝JST 4時間バケット（TTL 実効化）
**状態:** 実装済み（施策A）
**文脈:** AIキャッシュ TTL は4時間だが、キャッシュキーに毎時変わる `now`（"H時"）とスライドする `hourly[].t` が含まれ、キーが毎時変わって TTL が実質死んでいた。
**決定:** キャッシュキーを **JST 4時間バケット**（`bucketStartMs`）に固定する。バケット粒度は TTL と一致する4hを採用（当初案6hでなく）。
**理由:** キーの変動周期を TTL に合わせることでヒット率が上がり、コール数を最大24→6回/日・上限に抑えられる。
**結果:** `src/lib/aiCommentInput.ts` の `buildCommon()`。キャッシュは `/users/{uid}/aiComments/{hash}`（標準）と `c:{hash}`（カスタム）でプレフィックス分離。

## ADR-0016: AI プロンプトは「偵察役」スタンス＋事実ガードレール
**状態:** 実装済み
**文脈:** 農家は適期を逃せず「雨でも進めたい、注意点を知りたい」。作業可否を断ずるAIは現場に合わない。
**決定:** AIは審判でなく**判断材料を渡す「偵察役」**。作業の限界線（デッドライン）＋残るリスク＋軽減策を自然文で示し、決定は農家に委ねる。タスク別トーン: 畑しごと・施肥は攻め、農薬散布は慎重（撒き直しは使用回数を1回浪費する重い代償）。
**理由:** 現場ニーズと安全性の両立。命令形禁止・提案形限定・専門用語（VPD等）言い換え。
**結果（核心の線引き）:** 気象予報値（気温・降水量・降水確率・風速）は**データ厳守**。一方、乾き時間・飛散の風速目安などの**農学的一般知識はデータになくても一般論として使用可**（予報値のように断定はしない）。反捏造ルールと「乾き時間」の衝突を解消する核心。ai-comment/ai-custom で事実ガードレールは統一、スタンス系は ai-custom には入れない（ユーザー入力でスタンスが決まるため）。

## ADR-0017: AI アクセス制御＝サーバー検証＋env 許可名簿（B方式）・`aiAllowed` 単一フラグ
**状態:** 実装済み（将来 A2 移行予定）
**文脈:** AIコメントは有料相当機能。サーバー側でアクセス制御が要る。
**決定:** サーバー（Pages Function）が `jose` で Firebase ID トークンを検証し、env `AI_ALLOWLIST`（カンマ区切りメール名簿）で許可判定→401/403（B方式）。共有ガード `requireAiAccess`（`functions/api/_auth.ts`）。`/api/me` が `{aiAllowed}` を返し、クライアントは `aiAllowed` 1フラグで「実AI vs Coming Soon」「地点50/10」「CSV可否」を駆動。
**理由:** キー付与型（A方式）より env 名簿が実装最小で、決済導入前の限定公開に適する。
**結果（既知の負債）:** `aiAllowed` が「AI可否」と「プレミアム可否（地点上限・CSV）」を兼任＝**意味的過負荷**。現行1ティアでは正しいが、将来プラン分割（AIなし安価プラン等）で破綻する。課金フェーズで **A2（カスタムクレーム）へ移行**し、クレームに `tier` を載せて `isPremium`/`canUseAi` へ分離する（トークン検証の土台は再利用・捨て分ほぼゼロ）。**未設定だと全員401**（コスト安全側だが機能死）に注意。

## ADR-0018: AI コメントは未ログイン非表示・per-user キャッシュ
**状態:** 実装済み（共有キャッシュは保留）
**文脈:** AIはコスト源。未ログインに開放するとコストが漏れる。
**決定:** **未ログインユーザーには AIコメントを表示しない**（ログイン特典化＋ログイン誘導）。キャッシュは per-user（`/users/{uid}/aiComments/{hash}`）。
**理由:** `useAiComment` は uid 無しで comment=null＝既存挙動と整合。予報は地点共通なのに per-user だと重複生成するが、共有キャッシュ化（施策C）は現状ヒット率ほぼ0（キーが実質ユーザー固有）で中規模改修に見合わず**保留**。
**結果:** 共有キャッシュの成立条件は「座標グリッド丸め＋自由入力地点名の除外＋サーバー書き込み・クライアント読取専用」。有料化・地域集積が進んだ段階で再評価（Firestore TTL ポリシーが事実上必須）。

## ADR-0019: 収益化＝3ティア機能ゲート＋フリーミアム14日トライアル
**状態:** 3ティアゲートは実装済み／価格・トライアルは設計
**文脈:** 個人向け農業アプリの収益化。口コミ獲得が主経路で、クレカ強制は農村ユーザーに悪評リスク。
**決定:** アクセスを3ティア（未ログイン=現在地のみ / ログイン非許可=10件・Coming Soon / ログインAI許可=50件・実AI・CSV）でゲート。将来はフリーミアム＋**クレカ不要14日トライアル**、月¥300→¥500、年払い推奨。

## ADR-0020: セキュリティ構成（本人限定・キー別プロジェクト・CSP/_headers）
**状態:** 実装済み
**文脈:** 一般公開前のセキュリティ監査（2026-07-05）。攻撃面を体系的に洗い出した。
**決定:**
- **Firestore ルール:** 本人限定＋デフォルト拒否（`match /users/{userId}/{document=**} { allow read,write: if request.auth.uid==userId }`）。
- **APIキー分離:** Gemini キー（サーバー・別GCPプロジェクト・SAバインド・Gemini APIのみ・リファラー制限なし）と Firebase 公開キー（クライアント・HTTPリファラー制限あり・API制限なし）は別物。
- **セキュリティヘッダ:** `public/_headers` で全ルートに CSP・HSTS・X-Frame-Options:DENY・X-Content-Type-Options・Referrer-Policy・Permissions-Policy。
**理由:** 多層防御。CSP は「アプリの実通信先のみ許可」。
**結果（教訓）:** CSP許可リストは**ビルド済みバンドルの全外部ホストと突合**して検証すべき（手動洗い出しで Google Fonts と reCAPTCHA を見落とした）。`script-src` は起動ハンドラのため `'unsafe-inline'` 許容（XSS注入口が実在しないため実リスク限定）。**通信先を増減したら `_headers` 必更新。** Firestore ルールはコンソール管理でリポジトリ外＝リポジトリ化を推奨（[R-09](05-risk-register.md)）。

## ADR-0022: Firestore 書き込み規約（await+try/catch+UI／存在チェック＋直列化）
**状態:** 採用（教訓由来）
**文脈:** ①「設定変更が保存されない」silent fail バグ。②リロード後に保存済み設定が読めない競合バグ。
**決定:**
- Firestore 書き込みは**必ず `await` + `try/catch` + UIフィードバック**。fire-and-forget（await なし）は禁止。
- `ensureUserDocument` は「存在しなければ作る」だけ（毎回 setDoc しない）。
- 起動時初期化は `await ensureUserDocument` → `await Promise.all([loadLocations, loadUserSettings])` の順で**直列化**。
**理由:** ①await/catch が無いと書き込み失敗が UI に出ず、ユーザーは保存できたと誤認する。②毎回 `setDoc({createdAt}, {merge})` すると並行 getDoc に「createdAt のみ」の部分スナップショットを返す競合を起こす。並行できるのは読み込みのみ。
**結果:** `userRepository.ts` の `ensureUserDocument` と `App.tsx` の `onAuthStateChanged` 内順序が模範実装。

## ADR-0026: 日別予報は15日＋欠損日 `isPlaceholder`（16→15短縮）
**状態:** 実装済み
**文脈:** 日別予報の最遠日がモデル未計算タイミングで `temperature_2m_max/min` を null で返し、`?? 0` により 0℃ と誤表示された。
**決定:** ①取得・表示を `forecast_days=16→15` に短縮（最遠日を除去）。②気温 null の日を `isPlaceholder:true` にして「—」表示（`?? 0` 由来の0℃を構造的に遮断）。
**理由:** 二重防御。1日短縮で最もリスクの高い最遠日を落とし、null 防御で残るケースも根絶。`forecast_hours=384` は据え置き（15日目の夜間集計に翌日0-3時が要るため時間別は16日分残す）。
**結果:** プレースホルダー日は履歴タブで稼働中の既存描画パス（気温非表示＋各セル「—」）を再利用。AIコメント入力は既に `!isPlaceholder` で除外済み。分析タブの予報オーバーレイ4箇所にも除外ガードを追加。
