# 利用状況アナリティクス（有料化判断向け）設計

- 日付: 2026-06-29
- ステータス: 設計承認済み（実装計画待ち）
- 関連: [project_monetization](../../../../Users/kazma/.claude/projects/c--dev------/memory/project_monetization.md)（有料化戦略）

## 1. 目的

有料化を検討するにあたり、このアプリが **どの程度必要とされ、どのくらいの頻度で利用されているか** をざっくり把握する。地域分布は参考程度。

### 本質的な問い（再定義）
当初ユーザーが挙げた項目（LP訪問・各種クリック数）は「集客の入口」に偏っていた。有料化判断の核心は入口ではなく **「戻ってくるか・どれだけ使い込まれるか」= リテンションと利用頻度** である。本設計はこの再定義に基づき、リテンション/頻度/アクティブユーザー数を主役に据える。

### スコープ確定事項（ユーザー合意済み）
- 閲覧場所: **GA4の管理画面を直接見る**（analytics.google.com）。アプリ内に管理ダッシュボードは作らない。
- 同意処理: **プライバシーポリシーに記載のみ**。Cookie同意バナーは出さない（想定ユーザーは日本国内の農業関係者）。
- 登録地域: **参考程度**。常設ダッシュボードは作らず、オンデマンド集計スクリプトで対応。

## 2. 指標の優先度

| 優先 | 指標 | 取得元 | 当初項目との対応 |
|---|---|---|---|
| ★★★ | アクティブユーザー数（DAU/WAU/MAU） | GA4 自動 | 「天気情報の参照数」の正しい形 |
| ★★★ | リテンション（翌週/翌月も使う率・コホート） | GA4 自動 | （当初リストに欠けていた最重要指標） |
| ★★★ | 1人あたり利用頻度（セッション数/滞在） | GA4 自動 | 「どのくらいの頻度で」 |
| ★★ | ログイン数・ゲスト試用数（ファネル） | GA4 カスタムイベント | 「クリック数」 |
| ★★ | LP訪問数 | GA4 `page_view` 自動 | 「LP訪問者数」 |
| ★ | 登録地域の分布 | Firestore 集計スクリプト | 「登録地域」 |

## 3. アーキテクチャ

### ① GA4（Firebase Analytics）導入 ― 主役
- [src/lib/firebase.ts](../../../src/lib/firebase.ts) に `getAnalytics(app)` を追加。
- 環境変数 `VITE_FIREBASE_MEASUREMENT_ID` を新規追加（Firebase Console で取得 → Cloudflare Pages env へ登録）。
- これだけで GA4 が自動収集する: アクティブユーザー、リテンション（コホート表）、エンゲージメント（セッション頻度・滞在時間）、`page_view`（= LP訪問数）。
- 自前の集計・ダッシュボード実装は不要。Google提供の画面で閲覧する。

### ② カスタムイベント（最小4種） ― ファネル補強
- 薄いラッパー `src/lib/analytics.ts` を新設。`firebase/analytics` の `logEvent` を1関数で包み、analytics 未初期化時（measurementId 未設定のローカル開発等）は no-op にする。

| イベント名 | 発火箇所 | パラメータ | 意味 |
|---|---|---|---|
| `guest_start` | [LandingPage.tsx](../../../src/components/LandingPage.tsx) の「ログインせずに試す」クリック | なし | ゲスト試用数 |
| `login`（GA4予約イベント名） | ログイン成功時（LandingPage / LoginScreen） | `method: 'google'` | ログイン数 |
| `weather_view` | 天気データ表示到達時 | なし | コア機能利用 = 必要とされ度 |

- LP訪問数とクリックの大半は① の `page_view`/自動イベントで足りるため、カスタムは最小限に留める（YAGNI）。
- `weather_view` は **1セッション1回程度に絞る**。データ自動更新・再フェッチのたびに撃つと膨らみ、指標が歪む。セッション内で一度コア画面に到達したことが分かれば有料化判断には十分。

### ③ 登録地域の集計 ― オンデマンド、常設しない
- 管理用スクリプト `scripts/aggregate-regions.ts`。
- `firebase-admin` で全 `users/{uid}` 配下の地点データを読み、都道府県/地名で集計してコンソール出力。
- 月1回程度の手動実行を想定。ダッシュボード化しない。
- 注: サービスアカウント鍵が必要。鍵はリポジトリにコミットせず、ローカル環境変数 or gitignore 済みパスから読む。

### ④ プライバシーポリシー追記
- `/privacy-policy`（実体の所在は実装計画時に特定）に「本サービスはアクセス解析のため Google Analytics を使用します。Cookie等により匿名の利用データを収集します」旨の一文を追記。
- Cookie同意バナーは出さない。

## 4. データフロー

```
ユーザー操作 → analytics.ts(logEvent) → GA4 → analytics.google.com（閲覧）
Firestore(users / locations) → aggregate-regions.ts（手動実行） → コンソール出力
```

## 5. 環境・設定

- 追加環境変数: `VITE_FIREBASE_MEASUREMENT_ID`
  - ローカル `.env`（gitignore 済み）
  - Cloudflare Pages の本番 env
- Firebase Console で GA4 を有効化（プロパティ作成 or 既存連携）し measurementId を取得する作業はユーザーが実施。

## 6. エラー処理・非機能要件

- analytics ラッパーは measurementId 未設定時に no-op で安全に動作（ローカル開発でクラッシュさせない）。
- 計測失敗はアプリ本体の機能を一切ブロックしない（fire-and-forget で良い唯一の例外。理由: 計測はUXに影響せず、失敗時にユーザーへ通知する意味がないため。Firestore書き込みの安全則とは別カテゴリ）。
- GA4の数値は広告ブロッカー/Cookie拒否で実数より2〜4割欠損しうる。傾向把握には十分という前提で運用する（正確な実数は目的外）。

## 7. テスト・検証

- ローカル: measurementId を一時設定し、ブラウザのGA4 DebugView で `guest_start` / `login` / `weather_view` が発火することを確認。
- `aggregate-regions.ts`: テスト用 uid のデータで集計結果が正しいことを確認。
- 既存ビルド（`tsc -b && vite build`）が通ること。

## 8. リスク・注意点（盲点）

- **GA4は新規プロパティだとデータが空から始まり、リテンションは導入後からしか測れない。** 数週間〜1ヶ月寝かせて初めて有料化判断に使える数字になる。→ **「今すぐ入れて放置」が最も得**。導入が遅れるほど判断材料が遅れる。
- 既存ユーザーのリテンションは遡及できない（導入時点が起点）。
- 完全自前ログ方式（Firestoreイベントログ＋自作ダッシュボード）は本目的には過剰なため不採用。天気参照ごとのFirestore書き込みは課金増を招く。
